Main.sol Security Audit:

***Audit Summary***
    -This audit assesses the Main.sol contract functions, focusing on potential risks and mitigations.

    Audit Coverage:
    -Liquidity Pool Manipulation
    -Gas Cost of Looping
    -Code Analysis
    -Known Vulnerabilities
    -Solhint 
    -Mythril

    Key Findings:
    -No vulnerabilities were found that allow an attacker to steal funds from the TIFDAO Fund.
    -Some potential denial-of-service (DoS) risks exist but do not result in financial gain for attackers.



    1. Liquidity Pool Manipulation:
    Impact: Potential temporary disruption of sellTif() transactions.
    Risk: Minor

    Issue:
    -sellTif() relies on calculating the maxValuedToken dynamically. Since up to 100 tokens are included in the TIFDAO
    Index, it is not feasible to calculate all values on-chain efficiently.
    -If an attacker manipulates the liquidity pool, the calculated maxValuedToken's minTokenOutput for the swap may
    become inaccurate, causing transactions to fail.

    Mitigation: 
    -Liquidity self-corrects as arbitrageurs are incentivized to restore prices, ensuring sellTif() remains functional.
    -No direct financial gain exists for an attacker; the impact is only temporary.



    2. Gas Costs for _index Loops:
    Impact: Gas cost considerations for addTokenToIndex(), removeTokenFromIndex(), and sellRemovedToken().
    Risk: Minimal

    -The hard cap of 100 tokens ensures that even in the worst-case scenario, the loop costs ~220,000 gas, which is
    well below the 15M block gas limit.
    -Even under higher gas price conditions, the cost remains reasonable.



    3. Governance Risk: Malicious Token Addition
    Impact: Potential governance attack via voting in a paused token.
    Risk: Moderate, but no financial gain for attacker

    Issue:
    -Tokens that support pausing but do not implement paused() can bypass the current contract check.
    -If such a token is added to _index via DAO vote and later selected as minValuedToken or maxValuedToken while
    paused, contract operations may be disrupted.

    Mitigation:
    -The DAO must avoid voting in such tokens.
    -If a paused token is added maliciously (e.g., via flash loan attack on voting), users must vote it out.
    -The only potential gain for an attacker is causing temporary downtime.


    
    4. Front-End Integration Issue: callStatic & Contract Balance
    Impact: Incorrect ETH balance shown in callStatic calls
    Risk: Low

    Issue:
    -callStatic sends the simulated msg.value, causing incorrect contract balance readings.
    -Example: A test sending 0.01 ETH in 6 batches of 10 tokens resulted in a contract balance of 0.06 ETH instead of 0
    ETH. 

    Mitigation: 
    -Balance retrieval is now performed in a separate call, avoiding this issue.



    5. Price Feed Vulnerability: Reliance on a Single Data Source
    Impact: Potential delays or incorrect pricing due to API manipulation or downtime.
    Risk: Moderate

    Issue:
    -The contract originally used Chainlink oracles, but due to missing price feeds, it now relies on CMC API.
    -A single price source introduces risk:
        -Manipulated API responses could affect transactions.
        -Downtime on CMC API could delay contract operations.
    
    Mitigation:
    -Adding CoinGecko API as a second data source.
    -If prices from CMC & CoinGecko differ by more than 2%, transactions will fail to prevent manipulation.
    -Potential risk remains: a sophisticated attacker could compromise both APIs simultaneously (unlikely, but
    possible), which would result in the prevention of new transactions. There is no monetary incentive for the
    attacker in this case.



    6. Symbol Requirement for Token Addition:
    Impact: Prevents symbol spoofing for incorrect price lookups.
    Risk: Low

    Issue:
    -Tokens without symbol() cannot be added to _index to prevent users from manually submitting incorrect symbols.
    -Projects that rebrand may lose their original symbol, preventing price lookups.

    Mitigation: 
    -Ensures only legitimate tokens are added, preventing incorrect pricing data from APIs.
   


    7. Signing Server: Potential DoS Risk
    Impact: An attacker could overload the signing server by requesting excessive signatures.
    Risk: Moderate

    Issue:
    -An attacker can spam the signing server by repeatedly submitting slightly modified transactions to force the
    server to generate new signatures.

    Mitigation:
    -IP-based rate limiting on the web server.
    -Future enhancements may include session-based tracking to prevent VPN/proxy bypassing.



***Liquidity Pool Manipulation***
Summary: A threat agent would know the liquidity pool of the maxValuedToken calculated in the public
getMaxValuedToken(), and could alter the ratio of the liquidity pool to ensure that user trades from sellTif() do not
go through under the maximimum 2% price impact. This would be a temporary issue, as liquidity pool providers would be
monetarily incentivized to fix the ratio of the pool, and users purchasing TIFDAO Tokens would have the potential to
create a new maxValuedToken over time with new buyTif() transactions. Smaller user sellTif() transactions may still go
through safely in this instance, but this means that users could have periods where they are unable to sell the amount
of tokens that they want. There is an inherant bias in larger buyTif() transactions only being possible in larger
liquidity pools, which are typically more established and popular tokens, so the threat agent would likely have to
spend a large amount of money to manipulate the pool ratio. In such a case, their only gain would be pausing user
sellTif() transactions for a period of time. As sellTif() requires multiple calculations on the maxValuedToken it is
not feasible to make these calculations with the up to 100 tokens of the TIFDAO Index on-chain.



***Gas Cost of Looping Through _index***
Summary: In the worst-case scenario, if all 100 tokens in getActiveTokens() are active, the gas usage would be
~220,000 to loop through _index. This is within a reasonable price range and well within the targeted block limit of 15
million gas. The functions that loop through _index on-chain are addTokenToIndex(), removeTokenFromIndex(), and 
sellRemovedToken(). The breakdown is as follows:

Single Storage Read for _totalTokensInIndex
    Reading _totalTokensInIndex once at the beginning and caching it in memory avoids repeated storage reads.
    Gas cost: ~2,100 gas

Looping Costs with Memory Caching (100 tokens)
    Accessing the cached _totalTokensInIndex value in memory rather than storage. Memory accesses within a loop cost ~3
    gas each. 
    Gas cost: 100 × 3 = 300 gas

Array Push Cost (100 Active Tokens)
    Each push() operation for an active token to the array costs ~200 gas.
    Gas cost: 100 × 200 = 20,000 gas

Returning the Array (100 Items)
    Returning a large array incurs costs for memory allocation and data copying. Estimated ~2,000 gas per item for
    allocation and copy. 
    Gas cost: 100 × 2,000 = 200,000 gas

USD Cost of Execution
    At 50 gwei gas price (The average in 2024 was ~17 so this assumes a higher than average price) and ETH at $2,000:
    USD cost: 222,400 × 50 × 10^−9 × 2000 = $22.24
    (For reference, at 17 gwei, the cost would be ~$7.56.)

Does the function have the correct visibility?
    Yes, this function is meant to be called externally by the front-end.

Does this function have the correct access control?
    Yes, this function is meant to be called by anyone, however, a valid signature signed by the signing server must be

What does this function do? 
    This function is called by the front-end to purchase TIFDAO Tokens. It requires a valid signature signed by the
    signing server, and the minValuedTokenInfo and runningValueTotal calculated on the front-end. First the function
    checks that the timestamp of the users transaction is less than _lastTimestampCalled of the block buyTif() or
    sellTif() was called last, to ensure multiple transactions within the same block are not using the same input
    values which would generate incorrect minting values to the user. If timestamp < _lastTimestampCalled the error 
    OutdatedInputData() occurs. This function then calls _getMsgValue() to return the msg.value of the transaction. The
    swap _ethToMinTokenFromMsgValue() is then performed, where the 1.5% fee is taken and the Main.sol contract receives 
    the minValuedToken from the swap if the swap amount is greater than the minTokenOutput calculated in 
    getMinValuedToken().  _mintUserContribution() is then called to mint the tokens to the user based on the value of
    their added tokens vs the runningValueTotal of the tokens and the Ether balance of the Main.sol contract. A minimum 
    msg.value of 1000 Wei is required as the fee calculation divides the msg.value by 1000, otherwise the error 
    InsufficientInputAmount() occurs.

***Known Vulnerabilities: (https://scsfg.io/hackers/)***

ABI Hash Collisions:
    -Hashes of abi.encodePacked() are used in Validate.sol in the functions isValidSignatureBuyTif(),
    isValidSignatureSellTif(), isValidSignatureSellRemoved(), and isValidSignatureReinvest(). This is safe, however, as
    all of the values are of fixed length (either address, uint256, bytes32 and a fixed-length string of 28 bytes), and
    this vulnerability only arises when a hash calculation is used with packed ABI-encoded data that includes multiple 
    variable-length arguments. 

Ambiguous Evaluation Order: 
    -Main.sol and all imported contracts utilize Solidity version ^0.8.x. which enforces explicit evaluation order
    -Main.sol implements intermediate results and mostly deals with two values at a time through Utils.sol _multiply()
    and _divide(). For other values, parenthesis are used to ensure the operations are performed in the expected order,
    for example:
    uint256 availableMsgValue = ((985 * msgValue) / 1000);

Approval Vulnerabilities: 
    -Approval is required before _burnUserContribution() can call burnFrom() in sellTif() transactions. Only the amount
    required is approved, no infinite approval is used to protect the user from malicious spending if the contract is
    hacked. The cost of this to the user is having to call approve() every sellTif() transaction. The front-end 
    automatically calls approve() before sellTif().
    -See audit_Token.sol.txt for further information regarding approval frontrunning.

Exposed Data:
    -No sensitive data is used or stored on chain by any function in Main.sol, including private keys, signatures, or
    user data. Hashes of function parameters are stored in order to ensure protection against replay attacks. 

Frontrunning:
    1.Displacement Attack Scenario:
    -Attacker sees a pending buyTif() or sellTif() transaction.
    -They submit the same transaction with a higher gas price to execute first.
    -If successful, the original user’s transaction might fail or become irrelevant.

    Protections:
    -ethSignedMessageHash storing prevents replay once a transaction is executed, as the same hash cannot be reused.
    -The signing server only signs fresh transactions, so even if an attacker copies the data, they cannot generate a
    valid signature.
    -Using block.timestamp for _lastTimestampCalled ensures that any transaction with an older timestamp fails.

    Attack Outcome:
    -If the attacker executes first, the original transaction fails due to timestamp < _lastTimestampCalled.
    -They gain nothing because the victim must recalculate and get a new signature from the signing server.


    2. Insertion Attack Scenario (Sandwiching):
    -An attacker buys the token before a user's swap to artificially increase the price.
    -The swap executes at an inflated price, meaning the user overpays.
    -After the swap, the attacker sells at a profit, driving the price back down.

    Protections:
    -minTokenOutput ensures the TIFDAO Fund doesn't lose more than 2% of input value, rejecting manipulated swaps.
    -Using _lastTimestampCalled forces users to always execute the most up-to-date transaction.

    Attack Outcome:
    -Minimum output protection ensures the swap fails if price slippage exceeds 2%.
    -Attackers cannot profit because they cannot force the swap at a worse price.


    3. Suppression Attack Scenario:
    -An attacker spams high gas price transactions to delay user transaction execution.
    -If delayed too long, user transaction may expire before it executes.
    -This could force users to recalculate function parameters on the front-end before executing again.

    Protections:
    -The signing server already prevents transactions older than 10 minutes, meaning delays are limited.
    -Transactions using outdated data will fail if timestamp < _lastTimestampCalled, meaning no bad transactions get
    through.
    -EIP-1559 reduces block stuffing efficiency by burning base fees.

    Attack Outcome:
    -If gas prices spike or the network is congested, users may have to wait before executing.

Griefing Attacks:
    1. Transaction Griefing:
    -Attackers cannot submit the same transaction multiple times as the hash is stored to prevent this.

    2. Signature Spamming:
    -Attackers can overload the signing server by repeatedly submitting slightly modified transactions and force the
    signing server to issue new ones.
    -This will be dealt with on the server with IP-based rate limits.
    -Only hashes of valid signatures are stored.

    3. Gas Griefing:
    -_isTokenPaused() checks the return value to ensure success, otherwise false is returned.

Incorrect Parameter Order:
    -All function parameters are clearly defined and used in the expected order, including on the front-end.

Oracle Manipulation Attacks:
    1. Spot Price Manipulation (DEX-based):
    Vulnerability: 
    -If a smart contract relies on a single on-chain source (e.g., Uniswap spot price), attackers can use flash loans
    to manipulate the price for a single block, making tokens look much more expensive or cheaper.
    
    Protection:
    -On-chain prices are not used, quotes are fetched but are compared to a minimum output calculated from off-chain
    prices.

    2. Off-Chain Oracle Manipulation: 
    Vulnerability:
    -Currently using only CMC API is a vulnerability, so the new approach will be to also utilize CoinGecko's API and
    ensure the two prices are within 2% of eachother before averaging them together. If this is not the case, an error
    will be generated, as well as if only one price source is available. 
    
    Remaining Risk:
    -Both CMC and CoinGecko could be manipulated at the same time (unlikely but theoretically possible). The only gain
    to an attacker in this case would be delaying transactions to the Main.sol contract.
    -One of the two APIs might experience downtime, leading to delays in fetching prices.
    -A hacked API could return incorrect but still “realistic” prices within 2%.

Reentrancy Attacks:
    -State is changed before communicating with Uniswap contracts externally in sellTif(). It is not possible to do
    this in buyTif() as the swap needs to go through to calculate how many tokens the user added to the TIFDAO Fund
    with their purchase. This is not an issue as Uniswap v3 swaps do not trigger arbitrary external calls:
    
    1. buyTif():
    // Execute the external swap
    _ethToMinTokenFromMsgValue(msgValue, minValuedTokenInfo);
    
    // Mint after the swap 
    _mintUserContribution(msgValue, runningValueTotal);

    2. sellTif():
    // Burn first to prevent reentrancy
    _tifInstance.burnFrom(msg.sender, burnData.tifToBurn);

    // Now execute the external swap
    _maxTokenToEthFromMaxToken(
        burnData.minTokenOutput,
        burnData.maxTokenAmountToSell,
        maxValuedTokenInfo
    );

    3. _isTokenPaused() implements call():
    -It does not modify contract state.
    -It only reads from an external contract (paused()).
    -It returns a boolean value but does not trigger any follow-up transactions.
    -The external call (.call{ gas: GAS_LIMIT }) does not send ETH or allow reentrant execution.
    -Reentrancy is typically exploited when sending ETH (call.value() or .transfer()).
    -Since the function only queries the paused() state, an attacker cannot hijack execution flow.

    4. Cross-Contract Reentrency:
    -Main.sol does not rely on another contract's balance function to determine execution paths.
    -In BalanceScanner the balances are read before any action is taken.
    -quoteExactInputSingle() does not execute trades, only provides quotes.

Signature-related Attacks:
    -Signatures are used for buyTif(), sellTif(), sellRemovedToken(), and reinvestFundEther(). 

    Protection:
    -Ensuring that if erecover() returns address(0) it fails as it's compared against the signer's address which is
    hard-coded into Validate.sol.
    -All necessary transaction data is included in the signed message to prevent frontrunning attacks.
    -Signatures expire after 10 minutes to prevent users submitting data that is older than 10 minutes.
    -Signatures must have valid v and s values or the function reverts.
    -Replay attacks are prevented:
    if (usedSignatures[ethSignedMessageHash])
        revert SignatureUsedBefore();
    usedSignatures[ethSignedMessageHash] = true;
    -Cross-chain replay attacks are prevented as the project only runs on the Ethereum Mainnet.
    -EIP-2098 compact signatures can't be used as signature.length must equal 65 bytes.

Unexpected Ether Transfers:
    -This is not an issue as any additional Ether sent to the contract increases the value of the TIFDAO Tokens held by
    existing users. if 1 TIFDAO = 1 Ether and another Ether is sent then 1 TIFDAO = 2 Ether.
    -The contract doesn’t refund or execute unexpected transfers in response to received Ether.

Unprotected Swaps:
    -Each swap includes a minimumTokenOutput of 98% of the input value to ensure attackers cannot manipulate the price
    in a way that hurts the TIFDAO Fund. If the quote (which is compared against the minTokenOutput) is manipulated, it
    will not execute unless the quote is manipulated in the favor of the TIFDAO Fund:

    1. _ethToMinTokenFromMsgValue():
    amountOutMinimum: minValuedTokenInfo.minTokenOutput

    2. _maxTokenToEthFromMaxToken():
    amountOutMinimum: minTokenOutput

    3. _removedTokenToEthFromRemovedToken():
    amountOutMinimum: minTokenOutput

    4. _ethToMinTokenFromThisBalance():
    amountOutMinimum: minValuedTokenInfo.minTokenOutput

Address Validation:
    -Roles and contract assignments are completed in the constructor, or hard-coded if the values are known before
    deployment.
    -recoveredSigner == BACKEND_ADDRESS ensures signatures are only signed by the signing server.
    -address(0) cannot be added as a token in the TIFDAO Index due to checks in Governance.sol's propose().

Overflow & Underflow:
    -Solidity 0.8.x has built-in overflow & underflow protection, and unchecked{} is used where it is safe as described
    by the comments before unchecked blocks.

Self-Destruct:
    -selfdestruct() is not used anywhere in Main.sol

Insecure Randomness:
    -All calculations are deterministic.

Flash Loan Attacks:
    -DEX swaps all include a minimum output.

Storage Collisions:
    -Main.sol is not upgradeable so the storage layout does not change dynamically.
