Governance.sol Security Audit:

***Audit Summary***
    -This audit assessed Governance.sol and its modifications to inherited OpenZeppelin contract functions.
    -Modifications to functions work correctly, including additions of the InvalidOperation() error and the security
    checks added to propose(). This is shown within the tests from 'Function Testing' below.
    -Getting the token symbol directly from the token contract is more safe than letting users enter any token symbol
    they would like, as the token symbol is used to get the token price from the coinmarketcap API. If a token on 
    coinmarketcap has a different symbol from the token contract (from a rebranding, for example), then no price will 
    be fetched and the token will never be chosen as the minValuedToken. This is a better alternative than potentially
    making calculations based off of the wrong token's price if a user enters the wrong token symbol, whether
    maliciously or not. 
    -The code does not include any known vulnerabilities from https://scsfg.io/hackers/.
    -No vulnerabilities were found in Mythril.
    -After review, no security vulnerabilities were found in Governance.sol.

Function:   function propose(
                address[] memory targets,
                uint256[] memory values,
                bytes[] memory calldatas,
                string memory description
            ) 
                external 
                override(Governor) 
                returns (uint256) 

Does the function have the correct visibility?
    Yes, it's set to public just like the function it overrides as required by Solidity (can't change to external). 

Does the function have the correct access control?
    Yes, as I want anybody to be able to make a proposal. I have security features to control proposals mentioned in
    further detail below.

What does the function do?
    This function overrides the default Governor.sol propose() function to add the following security checks:
    -Ensures that only one operation is performed per proposal by checking that the targets.length == 1, otherwise an
    error InvalidProposalQuantity() occurs.
    -Ensures that the target of the proposal is the Main.sol contract address otherwise an error 
    InvalidProposalTarget() occurs.
    -Ensures that the function selector of the calldata is either addTokenToIndex() or removeTokenFromIndex(),
    otherwise an error InvalidFunctionSelector() occurs.
    -Ensures that the token to be added is not the address of the Token.sol contract or the WETH contract otherwise an
    error InvalidProposalToken() occurs.
    -Ensures that the proposed token to be added has a valid symbol() function, otherwise it reverts with
    InvalidTokenSymbol().
    -Ensures that any token added has a valid Uniswap quoter v1 quote otherwise an error InvalidUniswapQuote() occurs.



The functions votingDelay() and votingPeriod() are overrides of the IGovernor.sol interface, and set the required
voting delay and period. They are public functions used internally in Governor.sol, and can be queried externally.



The last three functions are required by Solidity. The first overrides quorum() of IGovernor.sol with
GovernorVotesQuorumFraction.sol. The next two override proposalDeadline() and _castVote() of Governor.sol with 
GovernorPreventLateQuorum.sol. I did not make any modifications to these functions.



***Known Vulnerabilities: (https://scsfg.io/hackers/)***

ABI Hash Collisions:
    -abi.encodePacked() is not utilized. 

Ambiguous Evaluation Order:
    -Governance.sol and all imported OpenZeppelin contracts utilize Solidity version ^0.8.x. which enforces explicit
    evaluation order.
    -Governance.sol does not utilize any complex expressions with multiple state-changing operations. 

Approval Vulnerabilities:
    -Governance.sol does not handle ERC-20 approvals, nor does it call approve() or transferFrom() which mitigates
    this risk.

Exposed Data:
    -Governance.sol does not use or store sensitive data on-chain. 

Frontrunning:
    -A malicious actor could frontrun a proposal and manipulate the Uniswap pool to create an incorrect quote, but this
    is not an issue as as the check is only to see if a quote is possible for the proposed token.

Griefing Attacks:
    -There are no timed elements in Governance.sol that can be manipulated by a threat agent.
    -Spam proposals are prevented due to the many security features that ensure only valid tokens can be proposed.

Incorrect Parameter Order:
    -All function parameters are clearly defined and used in the expected order.
    -The parameter order in abi.decode() matches the expected input, ensuring correctness.

Oracle Manipulation Attacks:
    -The Uniswap Quoter itself can be manipulated to provide incorrect quotes, but the function check only checks to
    see if quotes can be provided for the proposed token to be added.

Reentrancy Attacks: 
    -The only external contract call is to the Uniswap Quoter v1, and the address is hard coded in a constant variable.
    -call() and delegatecall() are not used, and no untrusted external contract interactions exist as state is not
    changed based on the external calls.

Signature-related Attacks:
    -Governance.sol does not implement signature-based voting, delegation, or execution beyond what is already included
    in OpenZeppelin’s GovernorVotes.sol.
    -There are no custom signature validation mechanisms in this contract, all vote counting and delegation rely on the
    ERC20Votes.sol logic, which securely manages votes using OpenZeppelin’s implementation.

Unexpected Ether Transfers: 
    -Governance.sol cannot receive Ether by normal means as receive() is blocked and there are no payable functions.
    -Any Ether maliciously sent would be locked, and no functions depend on Ether balances.

Unprotected Swaps:
    -None of the contracts handle token swaps, automated market making (AMM), or liquidity pools so there is no need
    for slippage protection.

Address Validation:
    -Proposals can only target the Main.sol contract address, and proposals cannot add TIFDAO or WETH to the index.
    -Proposed tokens must be contracts with a valid symbol() and Uniswap v3 liquidity pool.
